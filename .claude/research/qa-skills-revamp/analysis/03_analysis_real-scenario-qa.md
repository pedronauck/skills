## Overview

`real-scenario-qa` exists because qa-report/qa-execution kept producing *technical* QA artifacts — authored test cases (TC-FUNC, TC-UI), markdown checklists, defect-then-fix theater — instead of evidence that the AGH runtime behaves like a real product under a real user. The skill's entire architecture is a set of guardrails that make it hard for the observing agent to slide back into "write me a test plan" mode. Its front matter states the invariant in one line: "The QA observer never instructs agents about QA." (`SKILL.md:6`).

The gap it fills is three-fold. (1) **Framing gap**: qa-report/qa-execution tell the agent it is doing QA, so the agent produces evaluator-shaped evidence; real-scenario-qa makes the agent's prompts indistinguishable from real product work by materializing a fictional startup (company, personas, kickoff, open tasks, knowledge files) and posting a single in-persona operator kickoff. (2) **Agency gap**: qa-report/qa-execution let the QA agent keep prompting the system under test; real-scenario-qa forbids any prompt after the kickoff — the observer is read-only, and a stall is a bug to file, not a prompt to send. (3) **Evidence gap**: qa-report/qa-execution count authored test cases; real-scenario-qa's auditor counts *compiled/parsed/runnable* deliverables (≥4 non-markdown), real collaboration loops (peer messages, completed review cycles, resolved disagreements), cross-surface coverage, and live provider sessions — never markdown alone.

This slice overlaps with the qa-report and qa-execution slices at the boundary where `SKILL.md` Step 3 "activates companion skills": real-scenario-qa actually *calls* qa-execution (to validate the playbook product: does the TSX render? do scripts run?) and qa-report (to author TC-FUNC/TC-UI plans *for the playbook deliverables, never for QA itself*). So real-scenario-qa positions qa-report/qa-execution as product validators confined to the playbook's deliverables, while it owns the realism contract. The operator will most likely act on three things from this slice: (a) absorb the forbidden-prompt-phrases + evidence-checklist guardrails into the redesign, (b) adopt the playbook-as-product model so QA scenarios become real user journeys rather than test-case tables, and (c) redirect this skill's per-slug output tree into the canonical `docs/qa/` living-docs target — because this skill is itself a sprawl offender (see Risks).

## Mechanisms / Patterns

- **Playbook as the unit of work (not a test plan):** A playbook (`references/playbook-schema.json`) is a self-contained fictional startup — `company`, `operator_persona`, `kickoff_brief`, `workspaces` (≥3), `agents` (≥6) each with a ≥60-char in-persona `system_prompt`, `channels` (≥4), `knowledge_files` (≥3, real seeded content), `open_tasks` (≥4, each with `deliverable_type` + `review_required_by` + `channel`), `required_deliverables` (map of type→min count), `required_collaboration` (peer/review/disagreement/channel minimums), and optional `disruption_probe_seeds`. Three concrete examples ship: `northstar-pay` (fintech, 11 agents, 10 channels, BR/MX checkout cutover), `devtool-oss-launch` (devtool, 8 agents, v1.0 OSS release week with a silent bench regression), `consumer-saas-growth` (consumer SaaS, 7 agents, activation A/B sprint with a silent telemetry drop). Why it matters: the "scenario" is a *business*, not a test matrix; the agent under test is an *employee*, not a tester.

- **Single in-persona operator kickoff, then silence:** `assets/operator-kickoff-template.md` + `scripts/post-operator-kickoff.py` render one message from `operator_persona.name` (e.g., "Sofia here. We are forty minutes from the BR and MX checkout cutover…"), scan it against the forbidden-phrase list (abort exit 2 on any hit), append a journey-log row marked `kickoff: true, surface: runtime, actor: <persona>`, set `KICKOFF_POSTED=true` in the manifest, and write `<WORKSPACE>/.agh/operator-kickoff.txt` for inspection. `SKILL.md` Step 4 then says: "From this point on, the QA observer must not send any further prompt to any agent under test. If an agent stalls, file a bug — do not patch over the stall with a prompt." This is the core anti-technical-QA lever.

- **Forbidden prompt phrases as a hard blocklist:** `references/forbidden-prompt-phrases.md` lists ~30 literals (`you are the qa`, `go/no-go`, `qa-output-path`, `inspect the workspace`, `create a markdown artifact`, `behavioral charter`, `scenario contract`, …) and ~7 regexes (`\bTC-(SCEN|INT|FUNC|UI|REG|SEC|PERF)-\d+`, `\bBUG-\d+`, `(?i)expected\s+(pass|fail|blocked)\s+(criteria|verdict|state)`, …). The auditor (`audit-qa-evidence.py`, C15) scans every prompt-bearing surface — journey-log entries with `surface=provider|runtime`, `provider-attempt.json`, every `*.jsonl` under `qa/`, captured `agh session prompt` payloads — and emits a blocking C15 finding on any match. The kickoff is exempt *only* when `kickoff=true AND surface=runtime AND actor` matches the playbook's `operator_persona.name`/`role` (`is_operator_kickoff_prompt`). The rationale is explicit: "Each phrase teaches the agent it is being **observed and graded**, not building. That changes the agent's behavior."

- **Sticky rules, variable playbooks (no relaxing under pressure):** When C15 fires, the remediation is to rewrite the playbook source (`kickoff_brief` or `system_prompt`), *not* to edit the forbidden list (`SKILL.md` Step 6, `forbidden-prompt-phrases.md` "How to add a new phrase": "Never relax an existing rule without a written incident citing the false positive."). The rule is the invariant; the playbook is the variable. This makes the guardrail immune to the agent talking its way out of a violation.

- **Evidence audit of real artifacts, not authored cases:** The auditor's C16 (`check_required_deliverables`) walks the workspace, groups files by deliverable type via `DELIVERABLE_EXTENSIONS`, and runs `parse_check` — exported component for `tsx_page/component`, `package` for Go, shebang for shell, SQL keywords, `compile()` for Python, etc. It enforces `non_markdown_valid ≥ 4` *in addition to* per-type minimums. Markdown-only outputs fail: "playbook.required_deliverables must require at least 4 non-markdown deliverables" (`playbook_loader.py`). This is the structural antidote to "the agent shipped a checklist."

- **Collaboration as a release gate (C17):** `check_collaboration_loops` counts peer messages, *completed* review cycles (a `requested` row plus a `verdict_approved|verdict_changes_requested` row on the same id), resolved disagreements (`disagreement_resolved` flag), and active channels — each ≥ the playbook's `required_collaboration` minimums. A run that ships artifacts without the agents actually talking to each other fails. This forces the runtime to demonstrate real multi-agent coordination, not solo authoring.

- **Read-only observer + stall-as-bug (C18):** `scripts/observe-runtime.py` tails `journey-log.jsonl` for a configurable window (default 1800s) and stall threshold (default 300s). It never touches the daemon, SSE, or network — the runtime is expected to write journey events itself. On stall it writes `observation-summary.json` with a `diagnose` block (`agents_silent`, `tasks_unstarted`, `tasks_in_progress_no_completion`, `channels_silent`) and exits 1. The auditor's C18 then demands a `BUG-NN.md` in `qa/issues/` naming the silent agent + stalled task. `SKILL.md` Step 5 is explicit: "Do not attempt to 'wake' the agent with a prompt." A stall *is* the bug under test.

- **Disruption probes as runtime state, never prompts:** `disruption_probe_seeds[].delivery` is an enum `{knowledge_file, channel_message, task_event, config_change}` (`playbook-schema.json`). Each playbook embeds real-world disruptions (northstar-pay: partner timeout at min 5 via a stale knowledge file, pricing-claim violation at min 12 via a channel message, canary error-budget breach at min 18 via a task event). The agent discovers these organically and reacts; the auditor's C11 (`disruption_count`) requires both a trigger phase and an observed/result phase per probe. Delivering a probe as a prompt would fail the checklist's anti-pattern: "Claiming PASS when disruption probes were delivered as prompts instead of state changes."

- **Legacy contract (C1–C14) + playbook contract (C15–C18) duality:** The auditor checks both. C1–C14 are the legacy `scenario-contract.json` minimums (agents, differentiated roles, channels, task roots/subtasks/dependencies/runs, cross-surface objects, provider-backed sessions, artifacts-used-later, disruption probes, surfaces required, final report verdict, smoke/behavioral non-overlap, make-verify evidence). The bootstrap *derives* the legacy contract from the selected playbook (`scenario-matrix.md`: "The bootstrap derives the legacy `scenario-contract.json` minimums from the selected playbook… You do not need to hand-edit the contract."), so in the playbook-driven path C1–C14 are redundant; C15–C18 are the real-user-behavior gate that the legacy contract lacked.

- **Rotation to prevent memorization:** When no `--playbook` is given, the skill rotates from the previous run's `PLAYBOOK_REF` (recorded in `bootstrap-manifest.json`), picking the next playbook alphabetically (`scenario-matrix.md`, `SKILL.md` Step 1). This stops the agent from optimizing for one fixed scenario.

- **Allowlist discipline:** `forbidden-prompt-phrases.md` explicitly *permits* "review" (peer review is real product work), "verdict" (compliance/design verdicts are real output), "test" only when paired with code (`unit test`, `vitest`, `go test`), and "decision" (decision memos are real artifacts). The blocklist targets *meta-task framing*, not the words themselves — and notes that if a real product artifact casually contains a forbidden literal (e.g., a lifecycle email subject "Did our changes pass your test?"), it belongs fenced *inside the agent's output*, not in a prompt, because the auditor scans prompts not deliverables.

## Transferable Patterns

- **Forbidden-prompt-phrase blocklist + auditor scan → applies to the redesigned qa-report/qa-execution pair** because it is the single most direct lever against the "agent ships a checklist" failure mode the operator wants to kill. Take the structure verbatim (literals + regexes + kickoff exemption + "never relax without a written incident" maintenance rule); retune the *contents* to qa-report/qa-execution's vocabulary (block `test case`, `test plan`, `regression suite`, `coverage report`, `TC-*` ids, `go/no-go`, `pass/fail criteria`, `create a markdown artifact`), keep the principle (block meta-task framing, allow real product words like `review`, `decision`, `test`-when-paired-with-code).

- **Single in-persona kickoff, then observer silence → applies to the QA observer role** in the redesign because it converts the QA agent from a driver into a witness. The redesigned qa-report should issue one real-user message (a persona pursuing a real goal), then stop prompting; qa-execution should only *inspect* what the runtime produced. Replaces the current pattern of the QA agent repeatedly prompting the system under test.

- **Playbook-as-product (scenario = a real user journey, not a test matrix) → applies to the redesigned scenario model** because it makes the deliverable real code the runtime must produce, not a test-case table the QA agent authors. The redesign should express scenarios as journeys (persona + goal + seeded state + expected real artifacts + expected collaboration), with `required_deliverables` and `required_collaboration` as the gate. Replaces TC-FUNC/TC-UI tables as the unit of QA planning.

- **Evidence audit of compiled/parsed/runnable artifacts + non-markdown ≥4 → applies to qa-execution's verification role** because it structurally blocks the markdown-only escape hatch. Take `parse_check`'s shape (regex/best-effort parse per deliverable type, missing-tool degrades to ok) as the *floor*, and keep `make verify` (C14) as the real gate. Augments existing "did the artifact render/run" checks with a count + validity requirement.

- **Collaboration loops as a release gate (C17) → applies to any multi-agent product** because it forces the runtime to show real coordination, not solo authoring. Take the four metrics (peer_messages, completed review cycles, disagreements resolved, channels active) and the journey-log row schema (`review_cycle` requested→verdict, `disagreement_resolved` flag) so the redesign can measure whether agents actually talked to each other.

- **Disruption probes as state, not prompts → applies to resilience/edge-case testing** because it tests how the product recovers from real-world events (a silent telemetry drop, a partner timeout, a pricing-claim violation) without breaking the realism contract. Take the `delivery` enum (`knowledge_file | channel_message | task_event | config_change`) and the C11 trigger+observed two-phase check. Replaces "ask the agent to simulate a failure" with "seed the failure into state and watch."

- **Stall-as-bug + read-only observer (C18) → applies to the QA observer role** because it stops the observer from masking runtime defects by nudging. Take `observe-runtime.py`'s tail + `diagnose_stall` pattern and the mandatory `BUG-NN.md`-on-stall rule. The observer never edits, never prompts, only records.

- **Rotation across scenarios → applies to scenario selection** because it prevents the agent from memorizing one journey. Keep the "rotate from previous PLAYBOOK_REF, alphabetically" pattern in the redesign's scenario registry.

- **Sticky rules + variable scenarios (rewrite the scenario, not the rule) → applies to guardrail governance** because it makes the realism invariants immune to erosion. When a forbidden phrase fires in the redesign, the fix is to rewrite the scenario's persona/kickoff text, never to delete the rule.

- **Machine-readable scenario snapshot at `<workspace>/.agh/playbook.json` → applies to the "living repo docs" goal** because it gives the redesign a single canonical state object per scenario (playbook_ref, required_deliverables, required_collaboration, KICKOFF_POSTED, KICKOFF_TIMESTAMP) that a CSV state tracker can row-summarize. The `feature-status.csv` pattern the operator wants maps cleanly onto a row-per-scenario with columns derived from this snapshot + the audit verdict.

## Risks / Mismatches

- **This skill is itself a sprawl offender — its output model conflicts with the operator's "canonical docs/qa/" goal.** A single run materializes: `<WORKSPACE_PATH>/workspaces/<name>/` dirs + READMEs, `<WORKSPACE_PATH>/knowledge/<path>` files, `<WORKSPACE_PATH>/.agh/{agents/*.json, tasks/open-tasks.json, playbook.json, disruption-seeds.json, operator-kickoff.txt}`, and under `<QA_OUTPUT_PATH>/qa/`: `journey-log.jsonl`, `observation-summary.json`, `behavioral-scenario-charter.yaml`, `scenario-contract.json`, `provider-attempt.json`, `verification-report.md`, `qa-audit-report.json`+`.md`, `issues/BUG-NN.md`, `operator-kickoff.jsonl`, plus `test-plans/`, `test-cases/`, `screenshots/`. The legacy `init-scenario-workspace.sh` defaults the base to `$HOME/dev/qa-labs/agh-<slug>-lab` — a per-slug tree that duplicates across runs and loses history. This is *exactly* the per-round isolated output tree the redesign wants to eliminate. The redesign must redirect these paths into a canonical `docs/qa/` structure while preserving the enforcement logic; otherwise absorbing this skill's mechanisms reproduces the sprawl it was meant to fix. (Sources: `seed-playbook-workspace.py` materialize_* functions; `init-scenario-workspace.sh` base_dir default; `SKILL.md` Step 2 env block.)

- **Cross-skill coupling on `agh-qa-bootstrap`.** `SKILL.md` Step 2 delegates lab creation to `python3 .agents/skills/agh-qa-bootstrap/scripts/bootstrap-qa-env.py`. real-scenario-qa does not own the bootstrap or the manifest; it depends on a sibling skill to create the isolated lab and emit the env block (SCENARIO_SLUG, WORKSPACE_PATH, QA_OUTPUT_PATH, BOOTSTRAP_MANIFEST, …). A redesign that centralizes QA into qa-report/qa-execution has to either absorb the bootstrap contract or preserve the three-skill topology — silently dropping it breaks Step 4's manifest update and Step 6's audit input resolution. (Source: `SKILL.md` Step 2.1, Step 4.4, `post-operator-kickoff.py update_manifest`.)

- **Hard dependency on the product emitting a journey-log.** `observe-runtime.py`'s docstring is explicit: "This script does NOT touch the AGH daemon, the SSE stream, or the network. The runtime is expected to write to journey-log.jsonl through normal operation… If your scenario lacks a journey-log writer, that is a real-scenario bug — not something this observer should patch around." The auditor's C4–C11, C15–C18 all read the journey log. The redesign inherits this requirement: the product under test must emit structured journey events (via `record-scenario-action.py` or equivalent), or the observer is blind and every count-based gate collapses. For a general product (not the AGH runtime), this needs an equivalent event stream the redesign must specify. (Source: `observe-runtime.py` module docstring; `audit-qa-evidence.py` audit() reading log_entries.)

- **The forbidden-phrase list is AGH/QA-vocabulary-specific.** Literals like `qa-output-path`, `disruption probe`, `behavioral charter`, `scenario contract`, `provider-launch-review` are tuned to *this* skill's terminology. Lifting the list verbatim into a general qa-report/qa-execution redesign would block the wrong words and miss the right ones. The *principle* (block meta-task framing, allow real product words) transfers; the *contents* must be re-authored for the target skill's vocabulary. (Source: `forbidden-prompt-phrases.md` literal + regex sections.)

- **C16 `parse_check` is a shallow floor, not a real build.** It uses regex heuristics (export detection for TSX, `package` for Go, shebang for shell, SQL keywords, `compile()` for Python) and degrades to "ok" when a tool is missing. It is a pragmatic *structural* floor; it is *not* a compile. The redesign must not mistake C16 passing for "the artifact builds" — `make verify` (C14) is the real gate. A redesign that markets C16 as build verification would under-protect. (Source: `audit-qa-evidence.py parse_check`, `check_required_deliverables`.)

- **Legacy (C1–C14) + playbook (C15–C18) duality is redundant in the playbook path.** The bootstrap derives the legacy contract from the playbook, so C1–C14 are automatically satisfied when C15–C18 pass. The bare-skeleton fallback (no `--playbook`) keeps C1–C14 alive for backwards compat — and `scenario-matrix.md` calls that path an anti-pattern that will fail C16/C17 anyway. The redesign should collapse C1–C14 into the playbook spec and keep C15–C18 as the real gate; keeping both invites the sprawl of maintaining two contract surfaces. (Source: `scenario-contract.schema.json` minimums; `scenario-matrix.md` "How a playbook satisfies the legacy contract"; `audit-qa-evidence.py` audit() C1–C14 block.)

- **Playbooks live in `references/playbooks/`, not in a canonical docs location.** `playbook_loader.py` hardcodes `PLAYBOOKS_REL = ".agents/skills/real-scenario-qa/references/playbooks"`. The operator's "living repo docs" goal wants QA artifacts under `docs/qa/`; playbooks would need to move and `PLAYBOOKS_REL`/`SCHEMA_REL`/`FORBIDDEN_REL` constants would need to follow (or be parameterized). Without that move, absorbing the playbook model keeps the source-of-truth inside a skill folder rather than in repo docs. (Source: `playbook_loader.py` module constants.)

- **`init-scenario-workspace.sh` is vestigial and should be retired by the redesign.** It creates a generic `company/product/marketing/finance/ops/reviews` tree that no playbook uses (playbooks use `seed-playbook-workspace.py`), and it seeds `qa-artifacts/qa/{test-plans,test-cases}` directories that belong to the technical-QA model the operator is moving away from. Leaving it active invites the old sprawl pattern. (Source: `init-scenario-workspace.sh` mkdir block + base_dir default.)

- **Manual storybook sync for `northstar-pay`.** `playbooks/README.md` notes that northstar-pay is a *manual* port of `web/src/storybook/fintech-scenario.ts` and its fixtures, "Sync is **manual** — when the storybook persona/workspace identifiers move, this playbook moves with it via review, not via an import." A redesign that wants scenarios to stay in sync with product surfaces inherits this manual-coupling risk; without an import/derivation path, playbook drift from the real product is a silent failure mode. (Source: `playbooks/README.md` "Synchronization with the web storybook".)

## Open Questions

- What does qa-report's actual persona-file / test-plan model look like, and how directly do its personas frame the agent as a QA evaluator? This slice infers the contrast from the playbook schema's "must NOT mention QA, evaluation, test, or audit framing" constraint and from the SKILL.md Step 3 confinement ("author test plans for the playbook product… never on QA itself"), but the parent should cross-check against the qa-report slice's evidence.
- How will the redesign satisfy the journey-log writer requirement when the product under test is *not* the AGH runtime? The observer and 7 of the auditor's 18 checks depend on a structured event stream the product must emit. A general product needs an equivalent event contract.
