## Overview

This slice extracts the **standing constraints** any redesigned `qa-report` / `qa-execution` skill must respect, drawn from the AGH project memory under `docs/_memory/`. The memory is organized in three layers: a `standing_directives.md` file of perpetual postures (SD-001..SD-011), a `lessons/` registry of incident-validated rules (L-001..L-025, one file per lesson with an index in `README.md`), and a `_synthesis.md` that cross-references 8 forensic analyses into skill/lesson/CLAUDE.md candidates. The memory is explicitly a "review surface for Pedro to approve/reject before anything is generated as a real skill, lesson, or CLAUDE.md edit" (`_synthesis.md:5-7`), so its directives are the authoritative input for a QA skill redesign.

The slice overlaps with adjacent slices that map the **current** `qa-report`/`qa-execution` skill files and the AGH QA folder sprawl / `feature-status.csv` pattern. This slice's distinct contribution is the **hard constraints and vocabulary** the memory imposes: what "real-scenario QA" means in AGH terms, the QA home-policy and parallel-isolation rules, the forensic-first evidence standard, and the recorded convention for where QA artifacts live. It does **not** inspect the actual skill source files or the AGH QA folder layout — those belong to other slices — so any "sprawl" claim here is inferred from the memory's artifact-path citations, not from a directory listing.

The operator is most likely to act on three things from this slice: (1) the **vocabulary** the redesigned skills must speak ("real-scenario QA", "operator", "verification-report", "BUG-NNN", "home_policy" — and notably the memory never uses the word "dogfood"); (2) the **hard constraints** that have already been corrected once in the live QA skills (home policy, parallel isolation, forensic-first evidence) and so cannot be regressed; (3) the **class decision** a canonical committed `docs/qa/` with CSV state would represent versus the memory's recorded posture that per-round QA outputs and `.compozy/tasks/*/memory/` tracking artifacts stay local/unstaged.

## Mechanisms / Patterns

- **Standing Directive SD-005 — Real-Scenario QA Before Release** (`standing_directives.md:74-86`): a perpetual posture, not a per-task plan. It mandates (a) every program ends with a `qa-report` + `qa-execution` task pair, UI-bearing features include browser-based e2e; (b) the `real-scenario-qa` skill is the release validator and **delegates to** `qa-execution` + `qa-report` for inner mechanics; (c) hermetic QA still respects each provider's auth contract (bound-secret/brokered isolate `PROVIDER_HOME`; `native_cli` with `home_policy=operator` keeps operator `HOME`); (d) bug evidence (autonomy task_18 BUG-001/002/003, Hermes BUG-001..007) proves the QA pass surfaces bugs unit/lint/build coverage cannot. This is the single canonical statement of what "real-scenario QA" means and how the three QA skills layer.

- **Standing Directive SD-006 — Forensic-First Bug Fixes** (`standing_directives.md:89-101`): every bug-fix plan opens with a confirmed reproduction (timestamp, exact command, observed output) BEFORE listing changes; "I think"/"probably" is forbidden at the top of a fix plan. It also requires distinguishing symptom from root cause in writing, fixing at root cause, and re-running the narrow reproduction + impacted scenario + package tests. Encoded in `real-scenario-qa` and `cy-fix-reviews`. This is the evidence standard a redesigned `qa-report` must enforce.

- **Lesson L-007 — E2E harness regressions follow runtime contract changes** (`lessons/L-007-e2e-follows-runtime-contract.md`): the rule is "when a runtime contract changes (prompt augmenter, situation context, fixture format, manual-first UI state), the E2E mock and matchers ship in the same PR" (`:21-23`). Operationalization (`:27-33`): acpmock uses typed metadata not rendered-prompt substrings; Playwright shared wait helpers live in `web/e2e/fixtures/selectors.ts`; **"Real-scenario QA is the canonical regression net. `make verify` is necessary but not sufficient"** (`:29`); **"E2E regressions surfaced in the QA pass are NOT production bugs unless they reveal divergent runtime behavior. Fix the test infrastructure, not the runtime"** (`:33`). Anti-patterns (`:37-39`): `time.Sleep` to stabilize flaky specs, loosening matchers to substrings, skipping E2E because "unit tests cover it."

- **Lesson L-016 — Native Provider QA Must Respect Home Policy** (`lessons/L-016-native-provider-qa-home-policy.md`): the rule (`:40-60`) is to isolate **AGH runtime state** and **provider auth state** separately — always isolate `AGH_HOME`/ports/sockets for QA, but use `PROVIDER_HOME`/`PROVIDER_CODEX_HOME` only for `bound_secret`/brokered/explicit-isolated scenarios; for `native_cli` with `home_policy=operator`, preserve the operator's real `HOME`/native login state. "Never let a hermetic QA harness silently change the provider contract under test" (`:57-58`). Evidence (`:62-88`) shows this was already corrected across `agh-qa-bootstrap`, `qa-execution`, `real-scenario-qa` skills, `AGENTS.md`, `CLAUDE.md`, and the final QA plan — so a redesign MUST NOT regress blanket `HOME` isolation.

- **Lesson L-015 — Provider Auth Ownership Must Be Explicit** (`lessons/L-015-native-provider-auth-boundary.md`): the companion to L-016. `auth_mode = "native_cli" | "bound_secret" | "none"` declares auth ownership before secrets are considered (`:25-34`); `env_policy` and `home_policy` are part of the same boundary. "Do not add compatibility aliases or optional native slots to bridge the models; use one hard contract" (`:35-38`). A QA skill must read `auth_mode`/`home_policy` from the provider contract, not assume.

- **Lesson L-009 — Concurrent worktree/QA deadlock; isolate `AGH_HOME` + ports** (`lessons/L-009-concurrent-worktree-deadlock.md`): the rule (`:17-20`) — "When the user runs multiple AGH/Compozy agents in parallel worktrees, every test or QA run uses a unique `AGH_HOME` and a unique daemon port. Default home and default port use is forbidden in QA flows when concurrency is signaled." Operationalization (`:23-35`): `AGH_HOME=$(mktemp -d -t agh-XXXX)` or worktree-scoped home, unique daemon port, unique `tmux-bridge` socket, and **block any operation writing to `~/.agh/` or default ports when concurrency is signaled**. Detection signals (`:37-43`): `.git/index.lock`, `database is locked`, port collision, `cannot bind to socket`.

- **Lesson L-011 — "Fraco" test coverage pushback** (`lessons/L-011-fraco-test-coverage-pushback.md`): the single most consistent correction Pedro issues against generated `_tasks.md` (`:11-16`, BR-PT markers "fraco"/"leviano"). Rule (`:21-24`): count behaviors in the TechSpec and plan tests proportional to that count; reject 1-2 tests for many behaviors. Critical for QA redesign (`:36`): **"For QA-gating tasks (the trailing `qa-execution`): test cases enumerate every public surface touched, not a single smoke pass."** The lesson also (`:26`) authorizes "No new automated test" when an existing suite/lint/codegen/visual QA/documented manual evidence already owns the invariant — so QA density targets real-user surfaces, not blanket test-per-file.

- **Lesson L-004 — Manual = peer; E2E bookends** (`lessons/L-004-manual-equals-peer.md`): manual operator paths and autonomous paths converge on the same primitives; no separate queues (`:21-26`). Directly QA-relevant (`:24-30`): **"E2E coverage MUST include both manual-first bookends: (1) `user create → publish → coordinated execution` (2) `user-start session → direct prompt without coordinator`"** and "Operator UI must visually distinguish creation vs. publish/approval vs. run enqueue vs. coordinator spawn." A real-user QA scenario must cover both bookends, not only the autonomous lane.

- **Lesson L-018 — Delegated docs/runtime-truth audit** (`lessons/L-018-delegated-docs-runtime-truth-audit.md`): rule (`:30-34`) — every delegated docs run must be audited against runtime truth before the branch closes; "Treat any event name, route, outcome, status, CLI verb, config key, or UI literal that is not grep-able in the runtime as suspect." Anti-pattern (`:36-43`): **"Treating a delegated docs run's 'PASS' as evidence the prose is correct"** and inventing CLI verbs/routes/event names. Transferable to QA: a `qa-report` "PASS" is not evidence; QA claims must be auditable against the runtime/contracts.

- **Lesson L-014 — Sandbox vocabulary drift** (`lessons/L-014-sandbox-vocabulary-drift.md`): the sandbox feature was exposed as "environments" and "Once code, docs, generated references, and QA artifacts diverged, the feature became difficult to discover and easy to misrepresent" (`:13`). Pattern: QA artifacts that diverge from product vocabulary cause discoverability/misrepresentation harm → an argument **for** centralizing QA artifacts under a canonical, vocabulary-aligned location.

- **Glossary — Real-Scenario QA / Test Layers / `make verify`** (`glossary.md:273-292`): `make verify` is the blocking commit gate (`fmt → lint → test → boundaries → build`); Test Layers are Unit / Integration / E2E Runtime / E2E Web / E2E Nightly; **Real-Scenario QA** is defined (`:289-292`) as "the outer scenario orchestrator (`real-scenario-qa` skill) that builds a realistic multi-agent workspace and exercises CLI + Web + API surfaces end-to-end. Delegates to `qa-execution` and `qa-report` for inner mechanics." This is the canonical definition a redesign must match.

- **Synthesis findings on the QA skill stack** (`_synthesis.md`): Finding #3 (`:15`) lists the 5 rules repeated every backend session, including "(a) auto-append `$qa-report` + `$qa-execution` to every `_tasks.md`". Finding #6 (`:21`) — **"Three new skills exist locally and aren't in dispatch: `.agents/skills/real-scenario-qa/` (project-local). The pattern was distilled from autonomy `task_18` QA. ## Transferable Patterns

- **"Real-scenario QA" definition (glossary + SD-005) → goal (1) real-user dogfooding.** The memory already defines the target behavior precisely: build a realistic multi-agent/multi-channel/multi-task workspace and exercise **CLI + Web + API** end-to-end (glossary `:289-292`; SD-005 `:74-79`). A redesigned `qa-execution` should adopt this verbatim as its scenario-construction contract; it replaces "accumulate technical test cases" with "exercise the product the way a real operator does across all three surfaces."

- **L-007 "make verify necessary but not sufficient; E2E regressions ≠ production bugs" → goal (1).** The redesigned `qa-report` must (a) treat the QA pass as the canonical regression net for cross-component drift and (b) classify findings as either test-infra drift (fix the fixture/selector, same PR as the runtime-contract change) or a real production bug (L-007 `:29`, `:33`). This gives the report a built-in "is this a real-user-visible bug or a stale-test smell?" triage step that pushes QA toward real-user signal.

- **L-004 manual-first E2E bookends → goal (1).** A real-user scenario must cover both `user create → publish → coordinated execution` and `user-start session → direct prompt without coordinator` (L-004 `:24-30`). This prevents QA from collapsing onto only the autonomous lane and forces the two operator flows a real user actually uses.

- **L-011 "QA-gating tasks enumerate every public surface touched, not a single smoke pass" → goal (1).** The density rule for `qa-execution` is to enumerate the public surfaces (CLI verbs, HTTP/UDS endpoints, web routes, config keys) the feature exposes — i.e. the surfaces a real user would hit — not to pile up unit-style assertions (L-011 `:36`). This is the memory's antidote to "QA accumulating technical test cases."

- **L-018 "PASS is not evidence; grep-able runtime truth" → goal (2) living repo docs.** The docs-runtime-truth-audit pattern (treat any event name/route/CLI verb/UI literal not grep-able in the runtime as suspect; encode truthful phrasings as positive assertions in a committed test) transfers directly to a committed `docs/qa/` living doc: QA claims should be backed by grep-able runtime/contract citations, and truthful claims can be pinned by committed checklist tests (the `packages/site/lib/runtime-autonomy-docs.test.ts` model, L-018 `:30-43`).

- **Lessons-registry structure (`docs/_memory/lessons/README.md` index + one-file-per-lesson) → goal (2).** The memory's own "living repo doc" pattern is a concrete model for a canonical `docs/qa/`: an index table (ID | Title | Class) in a README plus one file per QA case/bug, each citing specific evidence (file path, commit, review issue, ledger entry) and explicitly **not** duplicating standing-directive rules (`lessons/README.md:1-50`). A `docs/qa/` with an index + per-case files would mirror this proven structure.

- **L-014 vocabulary-divergence warning → goal (2) centralization.** "Once code, docs, generated references, and QA artifacts diverged, the feature became difficult to discover and easy to misrepresent" (L-014 `:13`) is a direct argument for centralizing QA artifacts under a canonical, vocabulary-aligned location so QA state cannot drift from the product glossary. A `docs/qa/` that uses glossary terms (`real-scenario QA`, `verification-report`, `home_policy`) prevents the drift L-014 describes.

- **SD-011 agent-manageability → goal (2) CSV state.** "Every user-visible or operator-visible capability has an agent-manageability plan: CLI verbs with structured output, HTTP/UDS parity ... deterministic error contracts, discoverable status" means a QA-state tracking surface (CSV or otherwise) should be **agent-operable** (a CLI verb + structured output, not a web-only control) and should ship CLI/HTTP/UDS/config together in the same change. A `feature-status.csv` consumed only by humans would violate SD-011; one consumed via a `agh qa status` CLI verb with structured output would satisfy it.

- **L-009 parallel-isolation operationalization → both goals.** The concrete recipe (`AGH_HOME=$(mktemp -d -t agh-XXXX)`, unique daemon port, unique `tmux-bridge` socket, block writes to `~/.agh/` when concurrency signaled) is ready-to-paste guidance for a redesigned `qa-execution` bootstrap section (L-009 `:23-35`).

## Risks / Mismatches

- **Class barrier: a committed `docs/qa/` crosses the memory's "tracking artifacts stay local unstaged" line.** P-T2.27 (`_synthesis.md:331`) explicitly says `.compozy/tasks/_/memory/` files are **not** committed — "tracking artifacts stay local unstaged." The memory's recorded QA outputs (`.compozy/tasks/<slug>/qa/verification-report.md`, `issues/BUG-NNN.md`, the external `qa-labs/` lab) are all per-round/ephemeral or external, never committed living docs. A canonical committed `docs/qa/` with CSV state is a **new posture the memory does not prescribe and implicitly contrasts with** — it requires an explicit decision (likely a new standing directive, e.g. an SD-012) distinguishing "committed living QA doc" from "per-round tracking artifact." Without that, the redesigned skill would silently violate the recorded convention.

- **No hard directive mandates per-round QA trees, but the convention is entrenched in evidence.** L-007 (`:41-42`) and L-016 (`:75-76, :86`) consistently cite `.compozy/tasks/<slug>/qa/...` and `qa-labs/.../qa-artifacts/qa/verification-report.md`. SD-005 (`:80`) says "Every program ends with a `qa-report` task and a `qa-execution` task" — per-program, which is **compatible** with a living-docs model, but does not itself centralize. The operator's "centralize, stop duplicating cases across per-round trees" goal is **not contradicted** by a standing directive, but the memory treats QA reports as per-round outputs; replacing them with a single canonical `docs/qa/` is a greenfield hard-cut (SD-002 `:30-49`, L-006) that **must name its delete targets** (the old per-round `qa/` trees, the external `qa-labs/` convention) per the spec-authoring playbook (§1 "Greenfield-alpha = name delete targets").

- **Vocabulary mismatch: "dogfood" is not in the project memory.** A grep for `dogfood` across all of `docs/_memory/` returned zero hits. The glossary (`glossary.md`) is the authoritative vocabulary and uses "Real-Scenario QA." Introducing "dogfood" as a skill concept would be a glossary change the memory does not sanction; the redesigned skill should use "real-scenario QA" / "real-user scenario" to stay consistent with SD-005 and the glossary, or explicitly propose a glossary amendment.

- **L-016 home policy is a hard, already-corrected constraint — must not regress.** The memory records that blanket `HOME` isolation was already fixed across `agh-qa-bootstrap`, `qa-execution`, `real-scenario-qa`, `AGENTS.md`, `CLAUDE.md`, and the final QA plan (L-016 `:62-79`). A redesigned `qa-execution` that defaults to `HOME="$PROVIDER_HOME"` for all providers would re-introduce the false-blocker incident (L-016 `:9-22`). `home_policy` / `auth_mode` must be read from the provider contract (L-015 `:25-38`); AGH-state isolation and provider-auth isolation are separate.

- **L-009 parallel isolation is a hard constraint.** A redesigned `qa-execution` that reuses the default `~/.agh/` home or default daemon port when concurrency is signaled would deadlock (L-009 `:11-20`). The skill must block such operations, not merely warn.

- **SD-004 subagents read-only by default.** `qa-execution` typically runs as a delegated subagent. The memory is explicit: subagents "may write/edit/commit only when the parent's prompt explicitly delegates that action; otherwise the parent authors the change" (SD-004 `:68`). A redesigned QA skill that writes artifacts (living docs, CSV state) must respect this — the parent owns the write unless the dispatch explicitly delegates it. This is especially salient if the redesign moves QA outputs into a **committed** `docs/qa/` (a commit-grade action).
