## Overview

This slice quantifies the QA-artifact sprawl the redesign's goal (2) must absorb. The footprint is large and bifurcated: **~9.5 GB and ~181,600 files** spread across the AGH repo (`~1.39 GB / ~15,700 files`) and the out-of-tree `/Users/pedronauck/dev/qa-labs/` (`8.1 GB / 165,917 files`). Two clusters dominate disk: `final-qa/` at 1.2 GB / 13,336 files (a single pre-release master plan + one `_runs/` execution tree) and `qa-labs/` at 8.1 GB (30 isolated lab workspaces, two of which — `agh-tools-registry-task16-*` at 4.2 GB and `agh-final-qa-*` at 3.2 GB — alone hold 7.4 GB). The 29 `_archived/*/qa/` per-round trees total 168 MB / 1,881 files / 910 markdown docs. Everything else is small: `loops/qa/` (17 MB of codex-loop peer-review JSONL), the skill definitions (~0.5 MB combined), reference clusters (`openclaw/qa` 544 KB, `knowledge/test-and-qa` 5.1 MB), and `docs/ideas/qa-e2e/` (24 KB).

There are **five distinct artifact flavors**, not one. (1) The older "verification-report" flavor (hermes, bridge-adapters, ext-parity, core-tasks, sandbox, settings-ui, tasks-ui, etc.): `qa/{issues,logs,screenshots,test-cases,test-plans,verification-report.md}` with TC-* IDs prefixed by track (INT/SEC/FUNC/REG/UI). (2) The newer "behavioral-scenario-charter" flavor (network-threads, agent-soul, orch-improvs, the qa-labs): same tree plus `behavioral-scenario-charter.md`, `bug-reports/` (renamed from `issues/`), `evidence/`, `bootstrap-manifest.json`, and an explicit Agent Cast — this is the real-user dogfooding flavor the redesign wants to promote. (3) The "peer-review-of-specs" flavor (prod-ready, task-blocks, and the loops/qa cluster): `peer-review-{prompt,result,summary,incorporation}-roundN-*` files — this is codex-loop peer review of *design specs*, not runtime QA, and is a separate artifact system that should not be conflated with QA evidence. (4) The "final-qa master-plan" flavor: `_master-qa-plan.md` + 15 `_children/*.md` module plans (17,068 lines, 283 numbered scenarios) + `_runs/<run-id>/{screenshots,evidence,logs,scripts,issues}`. (5) The qa-labs "full-workspace" flavor: each lab is a simulated company (`product/`, `marketing/`, `ops/`, `finance/`, `company/`, `reviews/`, `knowledge/`) with `.agh/` runtime home and `qa-artifacts/qa/` mirroring the AGH-side tree.

The slice's most actionable finding for goal (2) is that **a canonical living-doc tracker already exists and works**: `.compozy/tasks/feature-stories/feature-status.csv` (254 rows, 17 columns, 224 KB, last modified 2026-06-22) tracks 253 feature stories with `qa_status`, `error_status`, `fix_status`, `retest_status`, `duplicate_or_overlap`, and an `evidence` column that points *into* the per-round `qa/evidence/batch-NNN-*/` and `qa/issues/BUG-NNN.md` paths. This CSV is the de-facto centralized state surface — but it is a single file in a feature-stories task dir, not a `docs/qa/` home, and its evidence pointers are round-relative (so they rot when rounds are archived). The redesign can generalize this pattern rather than invent one. Adjacent slices cover the `qa-report`/`qa-execution` skill internals and the `docs/_memory` QA-dogfooding specs in more depth; this slice owns the *where* and *how much*.

## Mechanisms / Patterns

- **Per-round isolated `qa/` tree (dominant pattern):** Every program/feature ends with a `<workflow>/qa/` tree written by the `qa-execution` skill (which generates `verification-report.md`) consuming `qa-report` artifacts (`test-plans/`, `test-cases/`, `issues/`, `screenshots/`). The shared output structure is documented verbatim in `.agents/skills/qa-report/SKILL.md` ("Shared Output Structure" block). After a program completes, its `<workflow>/` is moved into `.compozy/tasks/_archived/<id>-<slug>/`, so 29 historical `qa/` trees now live under `_archived/`. This is the root cause of the sprawl: each round reinvents the same tree with no cross-round index.

- **Two bug-directory names (`issues/` vs `bug-reports/`):** Older rounds (hermes, bridge-adapters, ext-parity) use `qa/issues/BUG-NNN-*.md`; newer real-scenario-qa rounds (network-threads) use `qa/bug-reports/BUG-NNN-*.md`. The `qa-report` skill spec says `issues/`; `real-scenario-qa` practice uses `bug-reports/`. This is silent schema drift a centralized layout must reconcile (pick one, alias the other).

- **Per-round BUG-NN and TC-* ID namespaces (duplication mechanism):** BUG- and TC-* IDs are scoped per round, not globally. `BUG-001` means "MCP OAuth config" in hermes, "session-event-query-finalization-race" in network-threads, and "unknown sandbox_ref returned 500" in feature-status.csv RT-007. Across `_archived/*/qa/` there are **85 distinct BUG- IDs** but `BUG-001` alone is referenced 168×, `BUG-002` 183×, `BUG-003` 74× — these are *different* bugs sharing an ID. Likewise `TC-INT-001` appears 171×, `TC-SEC-001` 177×, `TC-FUNC-001` 168× across rounds. There is no global bug or test-case registry, so cross-round regression tracking is impossible without a round prefix.

- **`feature-status.csv` living-doc tracking (the exception that works):** `.compozy/tasks/feature-stories/feature-status.csv` is a 254-row CSV with columns `story_id,source_slice,feature,actor,user_story,expected_behavior,entry_points,evidence,inventory_status,qa_status,error_status,documented_errors,fix_status,retest_status,duplicate_or_overlap,source_analysis,notes`. Each row cites `qa/evidence/batch-NNN-*/...` and `qa/issues/BUG-NNN.md` paths and a `source_analysis` pointer into `feature-stories/analysis/NN_analysis_*.md` (5 analysis files + summary.md exist). This is the only artifact that tracks QA state across rounds as a flat, greppable, diffable surface — exactly the pattern goal (2) wants, but stranded in a task dir.

- **`final-qa` master-plan + `_runs` execution split:** `.compozy/tasks/final-qa/` separates *durable planning* (`_master-qa-plan.md` 32 KB, `_children/01..15-*.md` 15 files 70-98 KB each, `_references/{openclaw,hermes}-qa-patterns.md`) from *per-run evidence* (`_runs/final-qa-20260503-044029-539204/{screenshots,evidence,logs,scripts,issues}` — the 1.2 GB bulk). The plan side is durable knowledge (17,068 lines, 283 numbered scenarios, 15 modules); the runs side is one-shot evidence. A centralized `docs/qa/` should absorb the plan side and *link to* (not copy) the runs side.

- **qa-labs full-workspace materialization:** `/Users/pedronauck/dev/qa-labs/` holds 30 isolated lab workspaces (8.1 GB), each a complete simulated company tree produced by `real-scenario-qa` via `agh/agh-qa-bootstrap/scripts/bootstrap-qa-env.py`. Each lab has `.agh/` (runtime home, `workspace.toml`, `playbook.json`, `disruption-seeds.json`, `agents/`, `tasks/`, `memory/`, `runtime/`), `.providers/`, `product/`, `marketing/`, `ops/`, `finance/`, `company/`, `reviews/`, `knowledge/`, and `qa-artifacts/qa/{test-cases,screenshots,test-plans,logs,notes,issues,verification-report.md,bootstrap-manifest.json,bootstrap.env}`. The newer `agh-qa-real-*` labs add `workspaces/{data-science,growth-lab,product-design,lifecycle-marketing}` — multi-workspace dogfooding. These labs are runtime scratch, not docs; centralization should *not* move them but should index their `qa-artifacts/qa/verification-report.md` verdicts.

- **`behavioral-scenario-charter.md` + Agent Cast (real-user dogfooding flavor):** Found in 4 archived trees and the qa-labs. The charter (see `network-threads/qa/behavioral-scenario-charter.md`) records Run ID, QA output path, bootstrap manifest path, lab workspace path, runtime home, API base URL, Operator Intent, Business Outcome, an Agent Cast table (Operator / Requester agent / Reviewer agent / QA observer), Provider Plan, Execution Matrix (SMOKE-001, TC-SCEN-001.., TC-INT-001, TC-UI-001, TC-REG-001), and Disruption Probes. This is the real-user dogfooding schema the redesign goal (1) wants to make canonical.

- **codex-loop peer-review as a *separate* artifact system:** `.compozy/tasks/loops/qa/` (17 MB) holds 10 rounds of `peer-review-{events,findings,incorporation,prompt,result,status}-roundN.{jsonl,md,err,txt}` — but this is peer review of the *Loops techspec* (`review_kind: techspec`, `readiness: NEEDS_REWORK`), not runtime QA evidence. The same peer-review machinery appears inside `_archived/1783210415046-e14d1ece-prod-ready/qa/` (62 blockers + 36 nits across 8 spec slices). These JSONL rounds are large (one is 5.2 MB, another 6.4 MB) and belong to the codex-loop tooling story, not the QA centralization story.

- **Auto-appended QA task pair + workflow extension:** `cy-tasks-tail-qa-pair` (`.agents/skills/cy-tasks-tail-qa-pair/SKILL.md`) auto-appends the canonical `$qa-report` + `$qa-execution` task pair to every `_tasks.md`, mirroring the Hermes template. `cy-qa-workflow` (`.compozy/extensions/cy-qa-workflow/extension.toml`) is a Compozy subprocess extension (binary at `/Users/pedronauck/Dev/compozy/looper/bin/cy-qa-workflow`) that wires QA into workflow runs via hooks `plan.pre_discover`, `plan.pre_resolve_task_runtime`, `agent.pre_session_create`, with `COMPOZY_QA_EVIDENCE_ROOT_TEMPLATE=".compozy/tasks/{workflow}"` and an audit command invoking `real-scenario-qa/scripts/audit-qa-evidence.py --strict`. These two are the *trigger* mechanism that guarantees every program spawns a per-round `qa/` tree — centralization must update `COMPOZY_QA_EVIDENCE_ROOT_TEMPLATE` or add an indirection.

- **Strict auditor as mechanical gate:** `real-scenario-qa/scripts/audit-qa-evidence.py` (39 KB) is invoked by `cy-qa-workflow` and by `real-scenario-qa` Step 6 with checks C15 (forbidden phrase), C16 (deliverable count), C17 (collaboration loop), C18 (stall), emitting `qa-audit-report.json`. This is the existing mechanical gate a centralized layout can reuse to enforce "no round closes without a tracker row."

## Transferable Patterns

- **`feature-status.csv` → canonical `docs/qa/state.csv` (or per-slice CSVs):** generalize the existing 17-column tracker. It already models `qa_status / error_status / fix_status / retest_status / duplicate_or_overlap` and points at evidence + source analysis. Move it (or a thin index of per-slice CSVs) under `docs/qa/` so it is version-controlled, reviewable, and survives archiving. This replaces the implicit "grep 29 archived trees" index with a flat, diffable surface. Applies to the whole QA estate because it is the only working cross-round tracker today.
- **`behavioral-scenario-charter.md` + Agent Cast → mandatory per-round front matter for goal (1):** promote the network-threads charter schema (Run ID, QA output path, bootstrap manifest, lab workspace, Operator Intent, Business Outcome, Agent Cast, Provider Plan, Execution Matrix, Disruption Probes) into the canonical per-round header. This makes "behave like a real user dogfooding" a structural requirement, not a prose aspiration. Replaces the older TC-only flavor (1) front matter.
- **`docs/ideas/qa-e2e/README.md` evidence standard → `docs/qa/` living checklist:** its 4-part evidence standard (action in UI/CLI + backend accepts without hidden errors + state visible in independent read path + survives refresh/deep-link/restart/resume) and "default assumption = product bug until disproven" rule are already a living doc. Promote `docs/ideas/qa-e2e/` to `docs/qa/playbook.md` as the durable home; it augments the per-round charter by defining what "validated" means across rounds.
- **L-019 (diagnostic data outlives primary record) → bug history must outlive rounds:** model bug reports as a separate durable registry keyed by a *stable global* BUG id, not the per-round `BUG-NN` that vanishes when a round is archived. The per-round tree is the "primary record" whose archival is routine; the bug history is the "diagnostic data" that must survive. A `docs/qa/bugs/` registry (or a `bugs.csv` column extension) absorbs this. Augments `feature-status.csv`'s `documented_errors` column.
- **`final-qa` master-plan + `_children` module decomposition → durable `docs/qa/plans/`:** the 15 module plans (01-daemon-boot … 15-observability) are durable knowledge worth a `docs/qa/plans/<module>.md` home, separate from the one-shot `_runs/` evidence. The split already exists inside `final-qa/`; centralization just relocates the plan side and links to (not copies) the runs side.
- **`audit-qa-evidence.py --strict` → mechanical gate that no round closes without a tracker row:** the existing C15-C18 auditor can be extended (or wrapped) with a check that refuses exit 0 unless the round's charter is registered in `docs/qa/state.csv`. Reuses the `cy-qa-workflow` hook seam (`COMPOZY_QA_AUDIT_COMMAND`) so no new trigger plumbing is needed.

## Risks / Mismatches

- **Moving `final-qa/_runs/` (1.2 GB) or qa-labs (8.1 GB) into `docs/qa/` would violate the "docs are lean, reviewable text" invariant** — `docs/qa/` must *index* one-shot evidence (screenshots, logs, HAR, JSONL) by path, not absorb it. The evidence is per-run scratch; the durable artifacts are the plans, charters, bug registry, and state CSV. Cite `docs/_memory/lessons/L-019` (diagnostic data outlives primary record, but lives in its own keyed store).
- **Per-round `BUG-NN` / `TC-*` ID collisions would silently corrupt a global registry if merged naively** — 85 distinct BUG- IDs but `BUG-001` is 168 *different* bugs. A centralized registry must mint global IDs (e.g. `<round-slug>-BUG-001` or a monotonic `BUG-NNNN`) and record the per-round alias, not just concatenate. Same for TC-* (TC-INT-001 is 171 different cases). This is the single biggest data-integrity risk in centralization.
- **Two bug-directory names (`issues/` vs `bug-reports/`) and two skill output conventions (`qa-report` spec vs `real-scenario-qa` practice) conflict** — a `docs/qa/` schema must pick one canonical name and treat the other as an alias, or the auditor and the tracker will disagree. The `qa-report` SKILL.md "Shared Output Structure" says `issues/`; `real-scenario-qa` charters use `bug-reports/`.
- **`loops/qa/` peer-review JSONL (17 MB) is NOT QA evidence — conflating it with runtime QA would pollute the central tracker** — `peer-review-findings-round1.md` shows `review_kind: techspec`, i.e. codex-loop peer review of design specs. It belongs to the codex-loop tooling story (alongside `.codex/loop/complex-deep-qa-real*`), not `docs/qa/`. A centralized QA home must explicitly scope what it *does not* own.
- **`agh-qa-bootstrap` lives at `.agents/skills/agh/agh-qa-bootstrap/` (nested under an `agh/` subdir) while `real-scenario-qa`, `qa-report`, `qa-execution` live at `.agents/skills/` top level** — any path rewrite in the redesign must account for the nesting or the bootstrap script invocation in `real-scenario-qa` Step 2 breaks. L-016 also cites this skill by a flat path that no longer resolves.
- **L-016 provider home policy must be preserved in any centralization** — the bootstrap contract (`PROVIDER_HOME` / `PROVIDER_CODEX_HOME` only for bound-secret/brokered/isolated-home lanes; native_cli operator-home preserved) is encoded in `agh-qa-bootstrap/references/bootstrap-contract.md`. A `docs/qa/` home that re-specs the lab bootstrap must not regress this, or native-provider QA will produce false blockers (the original L-016 incident).
- **Stale evidence paths in `docs/_memory/lessons/` are already a symptom of the sprawl** — L-007 cites `.compozy/tasks/autonomous/qa/verification-report.md` and `qa/issues/BUG-001.md`, but `autonomous/qa/` no longer exists (the program was archived/removed). Any centralization that moves per-round trees must either keep stable redirects or accept that lessons will rot. A `docs/qa/` home with a stable bug registry mitigates this for bugs but not for per-round evidence.
- **The 1.2 GB `final-qa/_runs` and the 3.2 GB `agh-final-qa-*` qa-lab duplicate the *same* run** — `_runs/final-qa-20260503-044029-539204/` and `qa-labs/agh-final-qa-20260503-044029-539204-lab/` share the run ID `20260503-044029-539204`. Centralization must decide whether the AGH-side `_runs` or the lab-side `qa-artifacts/qa/` is the canonical evidence home for a given round, or the duplication persists.

## Open Questions

- Is `agh-qa-bootstrap`'s nesting under `.agents/skills/agh/` intentional (a namespacing convention) or accidental drift? If intentional, should the other QA skills move under `agh/` too for consistency?
- Should `docs/qa/` absorb the `final-qa/_children/` module plans (durable) and *link to* `_runs/` (bulk), or should the entire `final-qa/` cluster be relocated wholesale and the split maintained inside `docs/qa/`?
