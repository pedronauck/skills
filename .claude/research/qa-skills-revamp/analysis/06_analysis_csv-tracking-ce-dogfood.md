# Analysis: csv-tracking-ce-dogfood

Read-only exploration of the slice `csv-tracking-ce-dogfood` (ordinal `06`) for the research prompt:

> Map QA skills context: qa-report/qa-execution skills structure, real-scenario-qa complementary skill, AGH QA folder sprawl, feature-status.csv tracking pattern, docs/_memory specs about QA dogfooding

## Scope

- Slice question: (a) How does `feature-status.csv` work as a living QA tracking doc — column schema, status lifecycle, how per-slice analysis files feed rows, how QA/fix/retest state is updated — and what would generalizing it into a canonical `docs/qa/` require? (b) What does the external `ce-dogfood` skill do better than qa-report/qa-execution for real-user dogfooding — workflow, report template, test-matrix taxonomy — and which pieces should be adopted?
- Primary sources: LOCAL `/Users/pedronauck/Dev/compozy/agh/.compozy/tasks/feature-stories/` (`feature-status.csv`, `analysis/`, `.dispatch/prompts/`); WEB `https://raw.githubusercontent.com/EveryInc/compound-engineering-plugin/main/skills/ce-dogfood/SKILL.md`, `.../references/dogfood-report-template.md`, `.../references/test-matrix-taxonomy.md`, and `https://thinkroom.kieranklaassen.com/d/njrS5TJhis`.
- Sources read in full vs. sampled: Read in full — `feature-status.csv` (all 253 rows, 17 columns, value distributions), `analysis/summary.md`, ce-dogfood `SKILL.md`, `dogfood-report-template.md`, `test-matrix-taxonomy.md`, the thinkroom article (via its API JSON, prose extracted). Sampled — `.dispatch/prompts/01_runtime-sessions.txt` (first 80 lines, to recover the CSV-maintenance contract), the `analysis/` directory listing, one CSV row (RT-001) as a lifecycle exemplar.

## Overview

This slice studies the two TARGET patterns the redesign wants to adopt: the living-CSV tracking pattern already in use inside AGH (goal 2 — centralize QA artifacts as living repo docs), and the external `ce-dogfood` skill (goal 1 — make QA behave like a real user dogfooding the product end-to-end). It overlaps with the adjacent `qa-report`/`qa-execution` slices only insofar as those are the skills being replaced; this slice does not analyze their internals, only contrasts them against ce-dogfood.

The AGH `feature-status.csv` is a 253-row, 17-column living tracker that was built by dispatching 5 parallel `explorer` agents (one per source-slice: `runtime-sessions`, `tasks-automation`, `extensibility-tools`, `network-bridges`, `memory-settings`) and aggregating their per-slice `analysis/NN_analysis_<slug>.md` "Candidate user stories" tables into one canonical spreadsheet. Its lifecycle columns — `inventory_status → qa_status → error_status + documented_errors → fix_status → retest_status` — encode the four-phase operator prompt verbatim (inventory → test & document errors → fix logistical/UX errors → retest post-fix). The `summary.md` explicitly freezes it as "the single working tracker" and forbids parallel trackers. BUG-NNN ids are the cross-reference key that ties a row's qa_status ("Passed after BUG-003 retest"), error_status ("BUG-003 resolved"), documented_errors, fix_status, and retest_status together, while the `evidence` cell packs `file:line` refs, `qa/evidence/batch-NNN-*/*.json`, and `qa/issues/BUG-NNN.md` paths into one field.

`ce-dogfood` is a single, opinionated skill that **merges** planning and execution into one workflow with a fix loop in the middle. Its seven phases (Scope → Analyze → Map+Matrix → Serve → Execute → Fix loop → Report) are diff-scoped (branch vs trunk, never whole-app) and drive a real browser via the `agent-browser` CLI. Its core discipline is **flows before matrix**: Phase 2 maps every user-visible change as Mermaid flowcharts of complete journeys (entry → action → branch points → side effects → true end state), then derives the test matrix by walking those flows. The thinkroom article frames the philosophy as "closing the verification loop" — the distance between a claim and its proof — and argues autonomous verification is "not agents being confident; it is agents being **auditable**, down to the commit SHA per scenario."

The operator is most likely to act on three things from this slice: (1) adopt ce-dogfood's flows-before-matrix discipline and test-matrix taxonomy to replace flat page checklists, (2) adopt ce-dogfood's fixed 6-value status enum to discipline the CSV's currently-drifting free-text status columns, and (3) keep the CSV as a persistent whole-product tracker while adding a `docs/qa/reports/` directory for per-run dogfood reports (the two serve different cadences and must not be collapsed).

## Mechanisms / Patterns

### feature-status.csv living-tracker pattern (LOCAL)

- **17-column lifecycle schema:** `story_id, source_slice, feature, actor, user_story, expected_behavior, entry_points, evidence, inventory_status, qa_status, error_status, documented_errors, fix_status, retest_status, duplicate_or_overlap, source_analysis, notes`. The first 8 are the inventory (what the feature is); the middle 5 are the QA lifecycle (status + error + fix + retest); the last 4 are cross-references and free-text. Each lifecycle column is a distinct phase of the operator's 4-phase prompt. (`feature-status.csv` header row)
- **source_slice prefix → row id:** `story_id` uses a 2-letter prefix per slice (`RT` runtime, `TA` tasks, `ET` extensibility, `NB` network, `MS` memory), so the row's origin slice is readable from the id. Distribution: TA 62, MS 52, ET 51, NB 46, RT 42 = 253. (`source_slice` column)
- **source_analysis column = the feed pipe:** every row carries the path of the per-slice analysis file that produced it (e.g. `.compozy/tasks/feature-stories/analysis/01_analysis_runtime-sessions.md`). The 5 analysis files are the upstream; the CSV is the roll-up. 42 rows point at file 01, 62 at 02, 51 at 03, 46 at 04, 52 at 05 — exactly matching the slice row counts.
- **Explorer-dispatch contract that builds the rows:** the `.dispatch/prompts/NN_*.txt` files instruct each explorer agent to include a "Candidate user stories" markdown table with columns `Suggested ID, Feature, Actor, User story, Expected behavior, Entry points, Evidence, Initial status, QA notes`, with `Initial status` constrained to one of `Inventory | Needs QA | Blocked by uncertainty | Potential UX/logistics risk`. The parent then aggregates those 5 tables into the CSV. (`01_runtime-sessions.txt` lines 40-80)
- **Row lifecycle, end to end (RT-001 exemplar):** `inventory_status="QA in progress"` → `qa_status="Passed after BUG-003 retest"` → `error_status="BUG-003 resolved"` → `documented_errors="BUG-003: failed pre-activation sessions were omitted..."` → `fix_status="Fixed"` → `retest_status="Retested: status health.failures updates immediately..."`. The BUG-NNN id is the join key threaded through 4 columns. (`feature-status.csv` row RT-001)
- **BUG-NNN cross-reference key:** 17 distinct bugs (BUG-001..BUG-017) are cited; BUG-007 most-referenced (10 rows), BUG-012 and BUG-014 next (6 each). 28 rows carry a BUG- id in `documented_errors`; 18 rows mention "after BUG" in `qa_status`. The id ties the row to a detail file under `qa/issues/BUG-NNN.md` (26 `qa/issues` citations across the CSV) and to evidence JSON under `qa/evidence/batch-NNN-<slug>/bugNNN-*.json` (476 `qa/evidence` citations).
- **duplicate_or_overlap dedup:** 21 of 253 rows carry a canonical-owner or overlap note (e.g. "Canonical owner for sandbox profile behavior; overlaps MS-030") so the testing loop doesn't double-test the same behavior. 232 rows leave it blank. The column is free-text, not a foreign key.
- **Free-text status drift (the anti-pattern):** `qa_status` has 33 distinct values; `retest_status` has ~40 distinct prose strings; `error_status` has ~25. Near-duplicate spellings proliferate: `No error` (137) vs `No product error` (69); `Not required` (143) vs `Not needed` (74); actor `Operator/Agent` (92) vs `Operator / Agent` (30) vs `Agent / Operator` (2). No enum, no validation.
- **Single-file merge surface:** the CSV is one 224KB file. Multiple agents updating it concurrently produce row-level (and sometimes cell-level) merge conflicts with no schema guard. There is no per-slice shard.

### ce-dogfood skill pattern (WEB)

- **Single skill merges planning + execution:** `ce-dogfood` is one skill whose 7 phases cover both matrix-building (Phase 2) and browser execution (Phase 4), with the fix loop (Phase 5) in the middle. This contrasts with the qa-report/qa-execution split where planning and execution are two separate skills. (`SKILL.md` workflow block, phases 0-6)
- **Diff-scoped, never whole-app:** Phase 0 refuses to run on the trunk ("no diff, nothing to dogfood") and resolves the trunk ref dynamically (`git symbolic-ref`, `gh repo view`, fallback `main`/`master`). A PR target is always diffable. (`SKILL.md` Phase 0 + Phase 1 bash)
- **Flows before matrix (the email rule):** Phase 2a maps every user-visible change as a Mermaid `flowchart` covering entry, action, branch points (validation error / empty / permission-denied), side effects (emails, jobs, notifications), and the true end state including click-through destinations. Phase 2b derives the matrix by walking those flowcharts. "An email sends" is not a pass — right recipient, right thread, scrolled to the right message is. (`SKILL.md` §Phase 2; `test-matrix-taxonomy.md` §1 Journeys)
- **Test-matrix taxonomy (5 dimensions):** (1) Journeys — Entry/Action/Result/Destination/Aftermath; (2) Functional checks — forms validate, links resolve, data round-trips, `agent-browser errors` clean, auth boundaries hold; (3) Experiential checks — vision alignment, copy, loading/transition states, persona paper cuts; (4) Edge/error/empty states; (5) Cross-cutting — responsiveness, accessibility, regression. Plus a "Mapping files to routes" table (views→pages, components→rendering pages, layouts→all pages). (`test-matrix-taxonomy.md` in full)
- **Two judges per scenario:** functional truth from the browser (instrument reading — `agent-browser snapshot`, `click`, `fill`, `errors`), experiential truth from a persona walk (re-reading the same run as each primary persona from `STRATEGY.md`/`VISION.md`). The persona pass hunts **paper cuts** — friction too small to fail a functional test but real enough to degrade the experience — each recorded with persona + severity. A **sharp** paper cut enters the fix loop like a failure. (`SKILL.md` Phase 4; article §4)
- **Fix-loop governor:** judge the size of the fix *before* touching code. Auto-fix only when small, well-understood, low-risk, contained to a few files, no schema/architecture/product trade-off. Escalate large/ambiguous changes to "Decisions for a Human" with what's broken, options+trade-offs, and a recommendation. (`SKILL.md` Phase 5; article §5)
- **Regression test per fix, one logical fix per commit:** every auto-fix ships a test that fails before and passes after; when no automated test is meaningful (pure copy/visual), a documented browser-replay stands in and the report states why. `ce-commit` scopes one logical fix per commit. After fixing, re-run the scenario AND adjacent journeys. (`SKILL.md` Phase 5)
- **Two terminal Blocked states:** `Blocked (human decision)` (from the governor) and `Blocked (needs human verify)` (for external-interaction legs the browser can't drive — OAuth, real email delivery, payments, SMS). Both are terminal for the loop; they wait on a person and are never silently re-queued. (`SKILL.md` Phase 4 + Phase 5; `dogfood-report-template.md` status legend)
- **Resumability via durable report doc:** the report is created at the *end of Phase 2* (as soon as the matrix exists) by instantiating `dogfood-report-template.md`, then updated incrementally after each scenario and each fix — not only at the end. The on-disk report is the source of truth for resume; the session task list mirrors it. Branch-slug rule: `<YYYY-MM-DD>-<branch-slug>-dogfood.md` where branch-slug lowercases and collapses non-alphanumerics to `-`. (`SKILL.md` Resumability + Phase 6)
- **Exit gate:** before declaring the branch ready, run the project's full automated test suite once. "A green matrix with a red suite is not ready." (`SKILL.md` Phase 5 end; article §5)
- **Orchestrator, not soloist:** delegates root-cause to `ce-debug`, commits to `ce-commit`, learnings to `ce-compound`, isolation to `ce-worktree`. (`SKILL.md` "Reusing Compound-Engineering Skills" table)
- **Report template (11 sections):** Diff Summary, Personas, Flows Tested (Mermaid), Test Matrix & Results (table: `# | Flow | Journey/Scenario | Status | Issue | Fix | Commit`), What Was Fixed (per issue: Symptom/Root cause/Fix/Regression test), Paper Cuts (by persona), Console Errors, Human Verifications, Decisions for a Human, Learnings, Final Status. Status legend: `Pending | Pass | Fixed | Skipped | Blocked (needs human verify) | Blocked (human decision)`. (`dogfood-report-template.md` in full)

### Philosophy (article)

- **The verification loop is the distance between a claim and its proof.** "Agents made building cheap. The cost moved: the expensive question is no longer 'can we ship this?' but 'does anyone actually know it works?'" CE answers that the loop itself is work an agent can do end-to-end, with people pulled in where judgment belongs to them. (article §1)
- **The most common way automated QA lies is by testing pages instead of journeys.** The breakage lives *between* the pages; flows-before-matrix is the correction. (article §3)
- **Independence budgeting.** A finding is only as trustworthy as the independence of whoever confirmed it. The browser's independence is physical (it cannot be talked into agreeing); the persona walk's is simulated (a second lens, not a second judge). "Dogfood buys independence in its instruments, not in its judgments; for the judgments, it buys a trail someone else can check." (article §6)
- **One loop, every altitude.** Regression test (red before, green after) → full suite (neighbors still pass) → independent review (quoted lines + fresh validators) → behavioral evals → dogfood (real browser walks real flows) → CI watch → residuals made durable. Each rung has the same grammar: claim plus independent proof. (article §7)
- **Autonomous verification is auditable, not confident.** "The loop never closes silently on an open question; it closes by making the question impossible to lose" — a Blocked row, a filed issue, a report section. Proof is durable down to the commit SHA per scenario. (article §7)

## Transferable Patterns

Aimed at goal 1 (real-user dogfooding) and goal 2 (centralize as living docs). Bullets, not a full design.

- **Flows before matrix (ce-dogfood Phase 2a)** → applies to the QA planning step because the AGH CSV rows are currently page/feature-grained, not journey-grained. Replacing the flat "one row per feature" inventory with Mermaid flowcharts of complete journeys (entry → action → branches → side effects → true end state) would catch the "email sends to wrong thread" class of bug that a per-feature row cannot express. Augments the `expected_behavior` column with a flow diagram per row.
- **Test-matrix taxonomy (5 dimensions)** → applies to the QA-execution coverage definition because it gives a checklist that forces journeys + functional + experiential + edge/empty + cross-cutting (responsiveness/accessibility/regression) coverage instead of "does the endpoint return 200". The current CSV's qa_status has no taxonomy behind it. Adopt as `docs/qa/taxonomy.md`.
- **Persona-driven experiential judging + paper cuts** → applies to the QA verdict because the CSV has only pass/fail/blocked; it has no concept of "passes but feels wrong for persona X." Add a `paper_cuts` column (or a linked per-run report section) recording persona + severity + fixed/deferred, so a row can be `Passed` AND carry sharp paper cuts routed to the fix loop. Replaces the binary qa_status.
- **Fix-loop governor (judge size before touching code)** → applies to the fix phase because the operator's prompt says "fix every logistical error or ux error" with no bound. The governor's rule (auto-fix only small/obvious/low-risk; escalate large/ambiguous/product-altering to a human) prevents the agent from making irreversible product changes just to clear a row. Adopt the two terminal states `Blocked (human decision)` and `Blocked (needs human verify)` in place of the CSV's single `Blocked`.
- **Regression-test-per-fix + one-logical-fix-per-commit** → applies to the fix phase because the CSV's `fix_status` currently records only "Fixed" with no proof. Adopt "every fix carries a red-before/green-after test (or a documented browser-replay with a stated reason)" and link the commit SHA from the row. This is the article's "auditable, not confident" principle made concrete.
- **Resumability via durable report doc created at matrix time** → applies to goal 2 because the CSV is durable but the QA *run* is not. Create a per-run report at `docs/qa/reports/<YYYY-MM-DD>-<scope>-dogfood.md` (mirroring ce-dogfood's `docs/dogfood-reports/`) the moment the matrix exists, update it incrementally after each scenario and fix. The CSV stays the persistent feature-state tracker; the report is the per-run checkpoint.
- **The 17-column lifecycle schema, generalized to `docs/qa/feature-status.csv`** → applies to goal 2 because the schema already encodes the 4-phase operator prompt and the BUG-NNN join key. Keep it, but discipline the free-text columns (see Risks). The `source_analysis` column pattern (each row links to its upstream analysis file) should become `docs/qa/analysis/NN_analysis_<slug>.md`.
- **Sketch of a combined `docs/qa/` layout:**
  - `docs/qa/feature-status.csv` — the living tracker (refined 17-col schema, disciplined enums).
  - `docs/qa/analysis/NN_analysis_<slug>.md` — per-slice deep dives that feed CSV rows (the existing pattern, relocated).
  - `docs/qa/reports/<YYYY-MM-DD>-<scope>-dogfood.md` — dogfood-style per-run reports (Mermaid flows, matrix table, fixes, paper cuts, Decisions for a Human, Final Status), adopted from ce-dogfood.
  - `docs/qa/issues/BUG-NNN.md` — per-bug detail (already exists as `qa/issues/`).
  - `docs/qa/evidence/batch-NNN-<slug>/*.json` — raw evidence (already exists as `qa/evidence/`).
  - `docs/qa/taxonomy.md` — the 5-dimension test-matrix taxonomy (adopted from ce-dogfood).
  - `docs/qa/report-template.md` — the per-run report template (adopted from ce-dogfood).

## Risks / Mismatches

- **Free-text status columns drift at scale.** `qa_status` has 33 distinct values, `retest_status` ~40, with near-duplicates (`No error`/`No product error`; `Not required`/`Not needed`; `Operator/Agent`/`Operator / Agent`/`Agent / Operator`). ce-dogfood uses a fixed 6-value enum (`Pending | Pass | Fixed | Skipped | Blocked (needs human verify) | Blocked (human decision)`). The CSV's drift violates the "auditable" principle — you cannot query "all blocked rows" reliably. **Fix:** adopt ce-dogfood's enum discipline for each lifecycle column and push prose into a dedicated `notes`/`evidence` cell.
- **Single 224KB CSV is a merge-conflict magnet.** Multiple agents editing one file concurrently produce row- and cell-level conflicts with no schema guard. ce-dogfood avoids this by giving each run its own report file. Constraint: the operator wants a *single canonical tracker*, so sharding risks re-fragmenting the sprawl the redesign is trying to kill. **Mitigation:** keep one CSV but define a serialization/merge convention (one row per line, sorted by story_id) and consider per-slice CSV shards with a generated roll-up view.
- **CSV is whole-product inventory; ce-dogfood is diff-scoped per-branch.** These serve different cadences and must NOT be collapsed into one file. A 253-row living tracker persists feature state across releases; a per-branch dogfood report tracks one pass. Mapping a dogfood run to CSV rows needs a *link* (e.g. a `last_dogfood_report` + `last_dogfood_verdict` column per row, or a join table), not a per-row report file. Forcing ce-dogfood's per-branch report shape onto the whole-product CSV would lose either persistence or diff-scope.
- **ce-dogfood's browser-driven flow requires `agent-browser` + a running dev server.** AGH QA already hit this wall: row RT-004's evidence cites `qa/evidence/batch-004-context-settings/rt039-rt041-browser-blocked.json` with "QA bootstrap browser.mode=blocked because neither browser-use plugin nor agent-browser CLI is available." 27 CSV rows are `Passed API/CLI where available; web blocked` and 2 are `Blocked by browser environment`. Adopting ce-dogfood's real-browser dogfooding is blocked until `agent-browser` is installed and a dev-server-start command (Phase 3) is wired for AGH. Without it, the redesign's goal 1 is partially inert.
- **ce-dogfood's autonomous fix-and-commit loop conflicts with the AGH explorer dispatch's read-only contract.** The inventory/QA explorer agents are scoped-write to one analysis file ("Do not run tests, build commands, package managers, git commands, or mutation commands. Do not fix code." — `01_runtime-sessions.txt`). ce-dogfood's Phase 5 does the opposite: applies fixes, adds tests, commits. The redesign must split QA (read-only, scoped-write to reports/CSV) from FIX (broader permissions, owns commits) as a separate agent phase — exactly the boundary ce-dogfood's governor already draws (auto-fix vs. escalate).
- **The CSV `evidence` cell packs too much.** It stuffs `file:line` refs + `qa/evidence/...json` paths + `qa/issues/BUG-NNN.md` into one free-text field. ce-dogfood separates these: the matrix table has distinct `Issue | Fix | Commit` columns; evidence lives as screenshots + `agent-browser errors`. Packing everything into one CSV cell makes the evidence unqueryable and merge-fragile. **Fix:** split into `evidence_paths` (structured), `bug_id` (foreign key), `fix_commit` (SHA) columns.
- **`duplicate_or_overlap` is free-text, not a foreign key.** 21 rows carry prose like "Canonical owner for sandbox profile behavior; overlaps MS-030" — readable by a human, unqueryable by a tool. ce-dogfood has no equivalent (it doesn't track inter-scenario overlap), so this is a net addition the redesign should keep but formalize as a `canonical_owner_id` + `overlaps[]` pair.
- **Persona walking is simulated, not independent.** The article is candid: "the persona walk's is simulated; it is a second lens, not a second judge, the same mind re-reading its own run." Adopting persona paper cuts adds experiential coverage but should not be oversold as a second independent judge — only the browser is physically independent. The redesign should pair persona-walk output with a real second pass (a fresh-context re-driver, per ce-dogfood's "behavioral evals" rung) for high-stakes journeys.

## Open Questions

- Is `agent-browser` installable in the AGH QA environment, and does AGH have a dev-server-start command (a `bin/dev`/`npm run dev` equivalent) that ce-dogfood Phase 3 can drive? RT-004 evidence says browser is currently blocked. This gates the entire real-browser dogfooding goal.
- Should the CSV be split into per-slice shards (one CSV per `source_slice`) to avoid merge conflicts, with a generated roll-up view? Or kept as one file with a disciplined serialization + merge convention? The operator wants a single canonical tracker, but 224KB single-file edits by many concurrent agents is fragile.
- What is the canonical status enum for each lifecycle column? ce-dogfood uses 6 values; AGH's qa_status has 33. The redesign needs to define the enum AND decide where the existing prose (e.g. "Passed after BUG-003 retest") migrates — likely a `notes` column with the BUG id as a foreign key.
- How should per-branch dogfood reports link back to CSV rows? A `last_dogfood_report` + `last_dogfood_verdict` column per row? A separate join table mapping report scenarios → story_ids? The two cadences (persistent tracker vs per-run report) need an explicit bridge.
- Does the operator want the fix loop (auto-fix + commit) to run inside the QA skill, or is fixing a separate agent phase with broader permissions — as the current explorer dispatch's read-only contract implies? ce-dogfood merges them; the AGH explorer contract separates them. This is a design decision, not a detail.
- Should "Decisions for a Human" entries from dogfood reports be promoted to CSV rows (with `fix_status=Blocked (human decision)`) or kept as report sections? ce-dogfood keeps them in the report; the CSV would need a way to surface them at the tracker level.
- The thinkroom article describes a "specialist prompt assets" persona-dispatch pattern (persona files seed generic subagents at dispatch time, no standing agents). Should the AGH redesign adopt that pattern for persona-walk dispatch, or keep persona-walk inline in the QA skill? The article argues the seeded-subagent shape avoids stale-registration/cache surprises (open question for the parent).
